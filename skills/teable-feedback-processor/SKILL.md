# Teable 反馈处理器

自动将用户反馈消息解析并同步到 Teable 表格。

## 工作流程

```
收到消息 → LLM 解析 (提取 ID/描述/反馈人/状态) → 查找现有记录 → 创建/更新表格
```

## 触发条件

当用户消息包含以下特征时触发此 skill：

- 包含问题反馈、bug 报告、需求建议等内容
- 包含类似 `Q001`、`Q002` 的问题编号
- 明确提到"反馈"、"问题"、"bug"、"建议"等关键词
- 用户直接要求"记录到表格"、"更新状态"等

## LLM 解析指令

调用此 skill 时，先让 LLM 将消息解析为以下 JSON 格式：

```json
{
  "id": "问题 ID（如 Q001，可为空）",
  "description": "问题详细描述",
  "reporter": "反馈人姓名",
  "status": "未解决 或 已解决"
}
```

**解析规则**：
- 从消息开头或明显位置提取 ID（如 `Q001`、`Q002`）
- 提取反馈人（如"张三反馈："、"from: 李四"）
- 根据语义判断状态（已解决/未解决）
- 剩余内容作为问题描述

## 使用脚本

核心处理脚本：`scripts/parse_and_update.py`

```bash
python scripts/parse_and_update.py "<原始消息>" "<LLM JSON 响应>" "[反馈人]"
```

**参数说明**：
- `<原始消息>`: 用户发送的原始消息
- `<LLM JSON 响应>`: LLM 解析后的 JSON 字符串
- `[反馈人]`: 可选，如果消息来源已知（如从聊天系统获取），可传入作为默认值

**返回结果**：
```json
{
  "action": "created" | "updated",
  "id": "记录 ID",
  "fields": {...}
}
```

## 示例场景

### 场景 1: 新问题反馈

**用户消息**:
> Q003 登录页面在移动端显示异常，按钮被遮挡

**LLM 解析**:
```json
{
  "id": "Q003",
  "description": "登录页面在移动端显示异常，按钮被遮挡",
  "reporter": "未知",
  "status": "未解决"
}
```

**执行**:
```bash
python scripts/parse_and_update.py "Q003 登录页面在移动端显示异常，按钮被遮挡" '{"id":"Q003","description":"登录页面在移动端显示异常，按钮被遮挡","reporter":"未知","status":"未解决"}'
```

### 场景 2: 状态更新

**用户消息**:
> Q001 已修复，测试通过

**LLM 解析**:
```json
{
  "id": "Q001",
  "description": "已修复，测试通过",
  "reporter": "未知",
  "status": "已解决"
}
```

**执行**: 脚本自动查找 Q001 记录并更新状态为"已解决"

### 场景 3: 带反馈人

**用户消息**:
> 来自王五的反馈：Q005 导出功能太慢，希望优化

**LLM 解析**:
```json
{
  "id": "Q005",
  "description": "导出功能太慢，希望优化",
  "reporter": "王五",
  "status": "未解决"
}
```

## API 配置

详见 `references/api-docs.md`

**关键配置**：
- Table ID: `tblJR0Vtfhc7wQcTi4i`
- 字段使用 `fieldKeyType=name`
- filter/orderBy 必须使用字段 ID

## 错误处理

1. **LLM 解析失败**: 使用默认值，将整个消息作为描述
2. **API 调用失败**: 返回错误信息，不阻断流程
3. **重复 ID**: 自动更新现有记录而非创建
